\# OPERATIONAL BASELINE REGISTER

\## NEXUS Sovereign Foundation v1

\## Controlled Phase 1 — Governed Operational Baseline Use



\### Purpose

This register defines the active operational baseline carried forward from the completed sandbox maturity track into Controlled Phase 1.



\### Baseline Status

Active for controlled operational use  

Internal governed baseline  

Not doctrine promotion  

Not production certification



\---



\## 1. Source-Precedence Baseline



\### Rule

Stronger governance sources outrank interface, summary, and reference surfaces.



\### Active baseline

Use the following precedence order for controlled phase work:



1\. canonical doctrine set

2\. `DOCTRINE\_IDENTITY\_REGISTRY.md`

3\. `KERNEL\_MANIFEST.json`

4\. `MASTER\_ARCHITECTURE.md`

5\. corrected observational outputs produced from stronger sources

6\. summary/interface surfaces only as non-authoritative reflections



\### Operational implication

No dashboard, export, or convenience surface may substitute for stronger governance or architecture sources.
Reviewable reference is not an authorized truth source.



\---



\## 2. Guardian Baseline



\### Rule

Guardian is required before governance-sensitive actions.



\### Active baseline

Guardian posture is:

\- manual internal preflight

\- advisory only

\- bounded checkpoint

\- non-approving

\- non-enforcing

\- non-mutating



\### Operational implication

Guardian must not be represented as autonomous approval, automated governance authority, or active enforcement.



\---



\## 3. Architecture / Lab Posture Baseline



\### Rule

Architecture governs intended lab posture.



\### Active baseline

For Controlled Phase 1:

\- `MASTER\_ARCHITECTURE.md` governs lab posture

\- inactive labs remain inactive unless explicitly re-admitted through governed review



\### Operational implication

Legacy or speculative lab states must not be inferred from dashboard or historical interface material.



\---



\## 4. Representation Baseline



\### Rule

Bounded wording is required unless provenance supports stronger claims.



\### Active baseline

Use bounded wording such as:

\- advisory

\- partial

\- under review

\- awaiting operator review



Do not use stronger finality-colored wording unless the surface also makes provenance, freshness, and validation basis explicit.



\### Operational implication

No unsupported certainty language should appear in operational summaries.



\---



\## 5. State-Integrity Baseline



\### Rule

Inactive reference surfaces stay out of truth flow.



\### Active baseline

The following are excluded from operational truth role:

\- `META\_CONTROL\_CENTER.md`

\- stale context export outputs

\- speculative dashboard/control-surface material

\- legacy labs as active architecture



\### Operational implication

These may remain reviewable as reference but not authoritative as current state.



\---



\## 6. Snapshot Baseline



\### Rule

Snapshots remain observational only.



\### Active baseline

Snapshots may:

\- summarize current state

\- support comparison

\- support review

\- carry telemetry-informed observations



Snapshots may not:

\- become canonical governance authority

\- override doctrine or architecture

\- silently inherit interface truth



\### Operational implication

Snapshot outputs must derive from stronger sources and remain explicitly observational.



\---



\## 7. Traceability Baseline



\### Rule

Decision and consolidation traceability are required for governed work.



\### Active baseline

Controlled Phase 1 work must preserve:

\- Decision Register discipline

\- Consolidation Log discipline where triggered

\- reconstructable action path

\- explicit residual uncertainty where present



\### Operational implication

Governed actions should be explainable after the fact without relying on memory or intuition alone.



\---



\## 8. Reversibility Baseline



\### Rule

Reversibility should be explicit where practical.



\### Active baseline

Where a governed action is reviewed or logged, the following should be stated when feasible:

\- reversibility class

\- rollback posture

\- impact radius

\- dependency risk



\### Operational implication

Reversibility is no longer treated as implied whenever explicit framing is possible.



\---



\## 9. Telemetry Baseline



\### Rule

Security/event telemetry may inform review in bounded form.



\### Active baseline

The following are active for controlled use in bounded form:

\- minimal structured event schema

\- bounded severity/tiering model

\- bounded response playbook

\- bounded trend interpretation



\### Operational implication

Telemetry may inform observability and security review, but does not create automatic escalation, doctrine, or authority.



\---



\## 10. Exclusion Baseline



\### Rule

Excluded materials remain excluded unless explicitly re-admitted.



\### Excluded from active baseline

\- `META\_CONTROL\_CENTER.md` as truth surface

\- legacy labs as active architecture

\- speculative dashboard/control-surface artifacts

\- stale export artifacts as authoritative state



\### Operational implication

Reviewable does not mean active.



\---



\## 11. Controlled Phase 1 Use Posture



\### Active use allowed

\- governed current-state summaries

\- bounded state review

\- Guardian preflight before governance-sensitive work

\- decision/consolidation traceability

\- bounded telemetry-informed review

\- bounded comparative snapshot use



\### Not active by default

\- broad runtime expansion

\- lab reactivation

\- doctrine promotion by momentum

\- interface-led authority claims

\- uncontrolled agent expansion



\---



\## 12. Boundary Note



This operational baseline register does not constitute:

\- doctrine promotion

\- canonical rewrite

\- production authorization

\- automatic authority expansion



It defines only the active governed baseline for Controlled Phase 1.
