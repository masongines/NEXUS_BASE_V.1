# 1) Core Definitions and Models

## 1.1 Knowledge Graph — System-Level Definition

A **knowledge graph (KG)** is a **persistent, structured state system** that encodes:

```
KG = (Entities, Relations, State, Time, Constraints)
```

* **Entities**: uniquely identifiable nodes (concepts, objects, agents, states)
* **Relations**: typed, directional edges with defined semantics
* **State**: current assertions about the system
* **Time**: validity intervals or event ordering
* **Constraints**: schema, rules, or invariants governing structure

### Boundary Conditions

A system qualifies as a KG if:

* relationships are **explicitly represented**
* traversal yields **deterministic structure**
* updates are **stateful and inspectable**

A vector index or embedding space alone **does not qualify**.

---

## 1.2 Functional KG Roles in Cognitive Systems

### A) Representational KG (Semantic Layer)

* Encodes **what is true or believed**
* Stable, schema-aligned
* Low mutation rate

---

### B) Memory KG (Temporal / Episodic Layer)

* Encodes **what happened and when**
* Append-dominant
* Time-indexed

---

### C) Execution KG (Operational Layer)

* Encodes **what is happening now**
* High mutation rate
* Used by agents for coordination

---

## 1.3 KG as a State Machine

A KG in a cognitive OS behaves as:

```
State(t+1) = State(t) + Δ(events, inferences, corrections)
```

Where:

* Δ includes both **observed events** and **derived relationships**
* correctness depends on **traceability of Δ**

---

## 1.4 KG vs Other Structures

| Property              | KG                | Relational DB      | Vector Index |
| --------------------- | ----------------- | ------------------ | ------------ |
| Relationships         | first-class       | implicit via joins | absent       |
| Time support          | native or layered | external           | absent       |
| Reasoning             | graph traversal   | query logic        | similarity   |
| Mutation traceability | explicit          | partial            | none         |

**SQLite‑Vector / embedded SQLite search**
SQLite‑Vector (and related SQLite vector extensions) embed vector search *inside* SQLite itself — no separate server, minimal memory, tiny footprint — flirting with the idea that brute‑force + good memory access *can be surprisingly effective* at modest dataset sizes. It runs everywhere SQLite does (Windows, WSL2, ARM, WASM) and brings high‑dim vector types + optimized distance functions into a local `.db` file. ([GitHub][1])

**LanceDB & Lance file format**
LanceDB is an *embedded, file‑native* vector store built atop the Lance columnar format. It’s designed for “laptop‑to‑petabyte” workloads, storing vectors and metadata, indexing with columnar efficiency, and exposing search via Python/JS APIs. Benchmarks outside your setup show *very low latency* and strong throughput when indexed on disk, and because persistence is native, “cold vs warm” latency effects matter a lot. ([GitHub][2])

**Qdrant single‑node OSS**
Qdrant is a full‑featured, open‑source vector database that runs as a local service (usable in WSL2 with disk persistence). It implements HNSW indexes, quantization, and performance knobs you can tune (e.g., memory vs I/O tradeoffs). Community and vendor benchmarks highlight stable recall with competitive latencies even at larger scales, though memory usage can be higher per workload versus ultra‑light embedded alternatives. ([Qdrant][3])

**Tradeoffs you’ll feel on NVMe**

* **Local vs server process:** SQLite‑Vector/LanceDB live in the same process as your app, so *cold start I/O and index loading* are read from disk, and you can measure insert/query p95 right alongside recall.
* **Index complexity:** Qdrant’s HNSW + quantization will outperform brute‑force for large sets (recall vs speed trade‑offs), but you’ll quantify *p95 latencies + throughput* under write load differently than you would with local file formats.
* **Persistent footprint:** All of these keep vectors on disk, but their *on‑disk storage sizes* after compaction + indexing differ — from tiny SQLite tables to larger HNSW graphs.
* **Recall measurement:** To anchor recall@k, you’ll likely compute a brute‑force ground truth once (e.g., via a NumPy/FAISS run) and then compare all approaches against it.

**Community bits & context**
People regularly debate these tools’ roles — lightweight embedded search vs purpose‑built vector servers — with tradeoffs around ease of setup, vector index maturity, and performance out of the box.

If you get stuck defining the exact metrics you want to capture (cold vs warm p95, memory footprint, recall variance), think of building these benchmarks *as a spectrum* — from pure in‑process brute force on disk to optimized ANN search with quantization and HNSW layering.

At the **smallest footprint end**, **SQLite‑Vector** is a cross‑platform SQLite extension that embeds vector search directly into your local DB with *no server process* — excellent for offline/edge where you want everything in a single binary and easy deployment. It’s specifically *serverless and in‑process* and perfect for semantic search without external services. However it’s under an *Elastic‑2.0‑style license* with restrictions on production/managed usage, so commercial caveats apply. ([GitHub][1])

A *complementary approach* — and closer to *columnar AI data tooling* — is **LanceDB**. It backs vectors and other multimodal data with a **Lance file format** designed for high I/O, nested schemas, and analytics workflows. It’s open (Apache‑2.0) and meant for local disk‑scale persistence without a heavy server, making join/SQL + vector search workflows plausible as you grow toward more analytical or graphy tooling. ([LanceDB][2])

On the other end of the spectrum is a **lightweight vector service** like **Qdrant**. This is a more mature, *disk‑backed vector engine* with HNSW ANN, quantization, and operational tuning designed for production. It runs as a service (REST/gRPC), so on Windows you typically host it inside WSL2 or a container, and it assumes tuned NVMe I/O for best tail latency. It’s also Apache‑2.0 licensed and easy to integrate with language clients. ([Medium][3])

In rough terms:

* **SQLite‑Vector** → smallest, in‑process, easiest to deploy (license caveats). ([GitHub][1])
* **LanceDB** → embedded but richer storage, strong on‑disk performance and future SQL/analytics/joins. ([LanceDB][2])
* **Qdrant** → standalone server, most matured vector engine with strong ANN features. ([Medium][3])

If you’re really pushing **local first with future graph style expansion**, the columnar / on‑disk vector store like LanceDB gives a smoother path toward analytical or hybrid workflows. ([LanceDB][2])

Researchers just released a reinforcement‑learning‑based reranker optimization method that reframes reranking as a sequential decision process, leading to better alignment between the documents selected and what an LLM actually needs to generate accurate answers — sidestepping the traditional reliance on static labels and human annotations. This suggests you can extract *usefulness* (not just surface relevance) with a reranker that reacts to LLM feedback. ([arXiv][1])

On the model side, the recently introduced **jina‑reranker‑v3** is a ~0.6B‑parameter multilingual listwise cross‑encoder reranker with a “last‑but‑late” interaction mechanism that scores whole sets of candidates jointly rather than encoding query/doc pairs independently. It’s competitive with much larger rerankers on benchmarks like BEIR and shows strong nDCG@10 performance, making it a practical sweet spot for local deployment if you want solid gains without huge GPU servers.([Jina AI][2])

Across independent benchmarks, adding a true reranker on *top‑k* lists (e.g., rerank the top 100 from your vector store) consistently boosts top‑10 retrieval metrics by noticeable margins compared to scoring by embeddings alone — something you can quantify with pytrec_eval or similar tools on fixed query sets.([AIMultiple][3])

Operationally, keep in mind cross‑encoders are heavier than bi‑encoders: they’re slower per pair but invaluable when you only need to rescore a small set of top candidates. Quantized weights (e.g., GGUF) and batching on GPU mitigate that, and local GGUF‑quantized versions exist so you can rerank entirely offline if you want reproducible, auditable pipelines. ([Hugging Face][4])

In practice many folks pull the top ~100 from their vector index, batch them through a local reranker, sort by that score, and then measure nDCG@10 with consistent seeds — keeping all artifacts and evaluation local. That’s a fast, high‑ROI step before exploring more complex integration work like knowledge graphs or RL‑optimized rerankers.

Here’s a simple, privacy‑first way to add “what’s going on under the hood?” observability to any LLM/agent app without sending data to a third‑party service.

---

# Zero‑SaaS observability for LLMs (drop‑in + local only)

**Goal:** trace every model call, tool invocation, and memory write—on your machine—so you can debug failures, measure latency/tokens/costs, and spot regressions, all without shipping sensitive data to the cloud.

## The picture (plain English)

* **Tracing:** like a flight recorder for your app. Each “span” is a timed event (e.g., `openai.chat`, `tool.weather.lookup`, `memory.upsert`).
* **OpenTelemetry (OTel):** a vendor‑neutral standard your code can emit to.
* **Collector + file exporter:** receive spans and write them as local JSON lines.
* **(Optional) Local UI:** run Langfuse or Helicone *self‑hosted* via Docker/Kubernetes to explore prompts, responses, and metrics.

## What you’ll instrument

Emit a span for:

* every **model call** (attributes: `llm.vendor`, `model`, `prompt_id`/`prompt_hash`, `token_count.input`, `token_count.output`, `cost.usd`, `cache.hit`, `outcome`),
* every **tool call** (`tool.name`, `args.size`, `outcome`, `latency.ms`),
* every **memory write** (`mem.key`, `mem.kind`, `size.bytes`, `outcome`).

## Quick start (OTel-only, no external UI)

1. **Run an OTel Collector that writes to disk**

```yaml
# save as otel-collector.yaml
receivers:
  otlp:
    protocols: { http: { endpoint: 0.0.0.0:4318 } }

processors:
  batch: {}
  attributes:
    actions:
      # redact emails and SSNs in any string attributes
      - key: pii.redactions
        action: insert
        value: "emails & ssns redacted"
exporters:
  file:
    path: ./traces/otel-spans.jsonl
    rotation:
      max_megabytes: 100
      max_days: 7

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [attributes, batch]
      exporters: [file]
```

Run:

```bash
docker run --rm -p 4318:4318 -v $PWD/otel-collector.yaml:/etc/otel.yaml -v $PWD/traces:/work \
  otel/opentelemetry-collector:latest --config=/etc/otel.yaml
```

2. **Emit spans from your app (Python example)**

```python
from opentelemetry import trace
from opentelemetry.exporter.otlp.proto.http.trace_exporter import OTLPSpanExporter
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor

# setup
provider = TracerProvider()
provider.add_span_processor(BatchSpanProcessor(OTLPSpanExporter(endpoint="http://localhost:4318/v1/traces")))
trace.set_tracer_provider(provider)
tr = trace.get_tracer("app")

def call_llm(prompt, model="gpt-4o-mini"):
    with tr.start_as_current_span("llm.call") as sp:
        sp.set_attribute("llm.vendor", "openai")
        sp.set_attribute("llm.model", model)
        sp.set_attribute("prompt.hash", hash(prompt))
        # ... do the real call ...
        # record results
        sp.set_attribute("token_count.input", 213)
        sp.set_attribute("token_count.output", 512)
        sp.set_attribute("outcome", "ok")
        return "...answer..."
```

3. **Lightweight redaction before export (defense‑in‑depth)**
   If you also log text attributes (e.g., prompts), apply a small scrubber **before** setting span attributes:

```python
import re
EMAIL = re.compile(r"[\w\.-]+@[\w\.-]+\.\w+")
SSN   = re.compile(r"\b\d{3}-\d{2}-\d{4}\b")

def redact(s: str) -> str:
    s = EMAIL.sub("[REDACTED_EMAIL]", s)
    s = SSN.sub("[REDACTED_SSN]", s)
    return s
```

* **Langfuse (self‑host):** Docker‑compose up a Postgres + UI. Point your app to Langfuse SDK *and* keep OTel for portability.
* **Helicone (self‑host):** reverse‑proxy style with nice per‑request analytics. Again, run via Docker locally.

* **Coverage:** ≥ 95% of model/tool/memory events produce a span.
* **Overhead:** < 5% added latency & CPU.

  * Verify with an A/B micro‑bench: run ~1,000 deterministic prompts **with and without** tracing; compare p50/p99 latency and CPU.
* **Basics visible at a glance:** per‑tool error rate, p50/p95 latency by model, tokens/$ per route, cache hit rate, and “slowest spans” list.

* Reject any span attribute larger than, say, **16 KB** (avoid giant prompt dumps).
* Keep **prompt hashes** in spans, not raw prompts (store raw only if you must, and after redaction).
* Rotate JSONL files and ship them to long‑term cold storage **you control** (encrypted disk).

Here’s a compact, practical way to add a **pre‑upgrade performance canary** for local OpenAI‑compatible servers (vLLM, Ollama, llama.cpp). It runs a tiny, deterministic load, measures latency/throughput/VRAM, records system traces, and only writes an artifact when there’s a *material* regression.

---

# What this is (in plain English)

Before you upgrade a model build, GPU driver, or server, run a quick, repeatable test that:

* Sends the **same prompt** with fixed decoding (greedy) and a fixed **random seed** to avoid noise.
* Measures **TTFB** (time to first token), **tokens/sec** (sustained), and **VRAM** usage.
* Captures a short **system trace** (ETW/WPR on Windows) around the run.
* Compares today vs. your saved baseline with **hysteresis + thresholds** (e.g., “flag only if TTFB slows by ≥12% for 2+ consecutive runs”).
* **Emits an artifact bundle** (CSV + JSONL + trace) only when a regression passes your significance window—so you don’t spam CI with noise.

It’s intentionally small: one Python runner, one Windows/WSL2 wrapper for ETW, one comparator.

---

# Why you’d want this

* **Catch slowdowns early**: spot driver/model/server changes that hurt latency or throughput.
* **Explainability**: ETW trace + metrics make regressions debuggable.
* **Noise‑resistant**: hysteresis + relative thresholds ignore normal jitter.
* **Privacy‑friendly**: all telemetry stays on disk locally via an **OTLP file exporter**—nothing leaves the host.

---

# Setup at a glance

**Targets (choose any):**

* vLLM OpenAI server (`/v1/chat/completions`)
* Ollama OpenAI server (enable REST endpoint)
* llama.cpp server (`--api` OpenAI‑style)

**Fixed run conditions:**

* Decoding: `temperature=0`, `top_p=1`, `frequency_penalty=0`, `presence_penalty=0`
* Seed: pinned (server permitting; otherwise encode determinism via prompt+greedy)
* Prompt: short but token‑rich; pin the system+user messages
* N requests: e.g., 8–16 per target to smooth variance
* Context: fixed `max_tokens` and input length
* GPU power mode locked if possible; run alone on the GPU

---

# Metrics to record

* **Wall‑clock**: TTFB (ms), total latency (ms), tokens/sec (sustained)
* **Server**: reported model, commit/ image tag, server version
* **Hardware**: GPU name, driver, CUDA/cuDNN (if applicable)
* **VRAM**: peak and average (via NVML or `nvidia-smi --query … --format=csv`)
* **Trace**: ETW session around the test (WPR profile for CPU/GPU/Disk)

# Example: Python runner (skeleton)

```python
# runner.py
import time, json, os, argparse, statistics
from pathlib import Path
import requests

def chat(url, model, prompt, max_tokens):
    t0 = time.perf_counter()
    r = requests.post(f"{url}/v1/chat/completions", json={
        "model": model,
        "messages": [{"role":"system","content":"You are a fast, literal assistant."},
                     {"role":"user","content": prompt}],
        "temperature": 0.0, "top_p": 1.0, "seed": 1234,
        "max_tokens": max_tokens, "stream": True
    }, timeout=120)
    r.raise_for_status()
    ttfb = None
    tok_count = 0
    start_stream = time.perf_counter()
    for line in r.iter_lines():
        if not line:
            continue
        if ttfb is None:
            ttfb = (time.perf_counter() - t0) * 1000.0
        # crude token counter from deltas (server-dependent)
        tok_count += 1
    total_ms = (time.perf_counter() - start_stream) * 1000.0
    tps = (tok_count / (total_ms/1000.0)) if tok_count else 0.0
    return {"ttfb_ms": ttfb, "total_ms": total_ms, "tokens": tok_count, "tps": tps}

def main():
    ap = argparse.ArgumentParser()
    ap.add_argument("--url", required=True)
    ap.add_argument("--model", required=True)
    ap.add_argument("--prompt", required=True)
    ap.add_argument("--n", type=int, default=8)
    ap.add_argument("--max_tokens", type=int, default=128)
    ap.add_argument("--out", default="run.jsonl")
    args = ap.parse_args()

    Path(args.out).parent.mkdir(parents=True, exist_ok=True)
    rows = []
    for _ in range(args.n):
        m = chat(args.url, args.model, args.prompt, args.max_tokens)
        m["ts"] = time.time()
        rows.append(m)
        with open(args.out, "a", encoding="utf-8") as f:
            f.write(json.dumps(m) + "\n")

    # quick summary to stdout (CI-friendly)
    ttfb_list = [r["ttfb_ms"] for r in rows]
    tps_list  = [r["tps"] for r in rows]
    print(json.dumps({
        "n": len(rows),
        "ttfb_ms_p50": statistics.median(ttfb_list),
        "ttfb_ms_p95": percentile(ttfb_list, 95),
        "tps_p50": statistics.median(tps_list),
        "tps_p05": percentile(tps_list, 5),
    }))

def percentile(vals, p):
    vals = sorted(vals); k = int(round((p/100)*(len(vals)-1)))
    return vals[k]

if __name__ == "__main__":
    main()
```

# Comparator (hysteresis + thresholds)

* Keep a rolling baseline JSONL (or CSV) per target.
* Compute deltas on **p50 TTFB**, **p95 TTFB**, **p50 TPS**, **p05 TPS**, and **VRAM peak**.
* Apply rules like:

  * Regressed if `p50 TTFB` slowed by **≥12%** *AND* `p95 TTFB` slowed by **≥8%** on two consecutive runs.
  * Or if `p05 TPS` fell by **≥10%** and `VRAM peak` increased by **≥8%** simultaneously.
* If regressed: write an **artifact bundle**:
  `artifacts/<date>_<target>/metrics.csv, metrics.jsonl, etw.etl, context.json`

*(Tip: store server/version/commit in `context.json` for bisects.)*

Here’s a fast, practical playbook for **stronger container/VM isolation on Windows**—what actually works today, what’s experimental, and what to try first.

---

### GO / NO‑GO (quick verdicts)

* **Hyper‑V–isolated Windows containers / micro‑VMs — GO.** First‑class on Windows; each container runs in a tiny VM with its own kernel (strong boundary). Use `--isolation=hyperv`. ([Microsoft Learn][1])
* **Firecracker (microVM) inside WSL2 — CONDITIONAL.** Possible with **nested virtualization** + WSL kernel tweaks; expect fragility and hardware/BIOS/.wslconfig prerequisites. Lab only. ([Gist][2])
* **Kata Containers + Firecracker — CONDITIONAL (mostly Linux).** Well‑supported on Linux; on Windows you’d run it *inside* WSL2 or another Linux layer (i.e., nested). ([GitHub][3])
* **gVisor/runsc on Windows — LIMITED.** Works via Docker Desktop’s WSL2 backend for experiments; not a native Windows kernel boundary and setup is fiddly. ([gVisor][4])

---

### What to implement **now** (minimal, auditable pattern)

**1) Use Hyper‑V isolation for any sensitive workloads.**

* Enable Hyper‑V and run Windows containers with `--isolation=hyperv` to get per‑container kernel isolation. Expect a small startup/memory tax for the stronger boundary. ([Microsoft Learn][1])

**2) Keep VMs off the external network by default.**

* Prefer **Internal/Private** virtual switches and only bridge to an External NIC when explicitly required. (Aligns with Hyper‑V isolation guidance and network hardening practices.) ([Microsoft Learn][5])

**3) Lock down the host with Windows Firewall rules (program‑scoped).**

* Create allow/deny rules **by executable path** for your Hyper‑V management tools and anything that talks to VMs:

  ```powershell
  New-NetFirewallRule -DisplayName "Allow Hyper-V Mgmt" -Program "C:\Windows\System32\virtmgmt.exe" -Direction Outbound -Action Allow
  ```

  Audit with `Get‑NetFirewallRule`. ([Microsoft Learn][6])

**4) Enforce “deny‑by‑default” on executables with WDAC (App Control).**

* Start with an allow‑list policy that permits only your signed management tooling and required Hyper‑V components; iterate as you see blocks. (Intune “App Control for Business”/managed installer can help scale approvals.) ([Microsoft Learn][7])

**5) Capture an audit trail per run.**

* Record: host+BIOS snapshot, VM image hash, WDAC policy hash, exported firewall rules, and begin/end markers in an ETW/WPR trace for VM lifecycle actions. (These align with Windows firewall/WDAC operational practices.) ([Microsoft Learn][8])

---

### If you *must* try Firecracker on Windows

* Do it **inside WSL2** only after confirming:

  * `nestedVirtualization=true` in `%USERPROFILE%\.wslconfig`,
  * `/dev/kvm` exists, user in `kvm` group, and your hardware/BIOS exposes VMX/SVM.
    Expect to patch/use a WSL kernel with KVM enabled; community guides exist but are version‑sensitive. Use for lab density/latency experiments—**not** your primary boundary. ([Gist][2])

---

### Notes on gVisor / runsc

* You can attach `runsc` as an extra runtime **inside Docker Desktop’s WSL2 VM** for testing. It improves Linux‑side isolation but isn’t a Windows‑native kernel boundary, and integration on Windows is still evolving. ([gVisor][4])

Here’s a compact, Windows‑first recipe to make your ML model files (GGUF or safetensors) **verifiable end‑to‑end**—so pulls are tamper‑evident and loaders can reject unsigned artifacts at load time.

---

# Why this matters (quickly)

* **Supply‑chain attacks** can swap weights, quantizers, or licenses without you noticing.
* **Signing + provenance + SBOM** → anyone can verify what they’re loading, where it came from, and under what license.

---

# The minimal blueprint

## 1) Content hashing with a Merkle tree

* Split the model file into fixed chunks (e.g., 4–16 MiB).
* Hash each chunk (SHA‑256), then hash up to a single **Merkle root**.
* Store:

  * `chunks.json` (offsets, sizes, per‑chunk hashes)
  * `merkle_root.txt` (just the root)

> Why: lets you prove large files match exactly, and enables partial verification for resumed downloads.

## 2) Sign the artifacts with Sigstore/cosign

* Sign the **weight file**, **merkle_root**, and the **manifest** (see step 4) using **cosign** keyless or key‑based.
* Publish/attach the Sigstore **transparency bundle** (Fulcio cert + Rekor entry) next to your release.

Example (PowerShell):

```powershell
cosign sign-blob model.gguf --output-signature model.gguf.sig --bundle model.gguf.sigstore.bundle
cosign sign-blob merkle_root.txt --output-signature merkle_root.txt.sig --bundle merkle_root.sigstore.bundle
```

## 3) Emit an ML‑aware SBOM (CycloneDX or SPDX)

* Produce `model.bom.json` that includes:

  * **File hashes** (model, chunks.json, tools used)
  * **License** (SPDX ID), source URLs, dataset notes
  * **Dependencies** (tokenizer files, custom ops, loader runtime)
* Keep it machine‑readable and versioned (CycloneDX “ML‑BOM” profile or SPDX with custom fields).

## 4) Capture deterministic quantization provenance

Record **exactly** how the file was produced:

* Quantizer tool name + **git commit** and **binary hash**
* Parameters (bits, group size, symmetric/asymmetric, etc.)
* RNG seed, calibration data/table checksums
* Input → output file hashes

Store this as `provenance.json` and **sign it** with cosign as above.
If using **GGUF**, also mirror key items in GGUF metadata (custom kvs) for at‑load display.

## 5) Bundle layout (ship these together)

```
model/
  model.gguf
  model.gguf.sig
  model.gguf.sigstore.bundle
  chunks.json
  merkle_root.txt
  merkle_root.txt.sig
  merkle_root.sigstore.bundle
  provenance.json
  provenance.json.sig
  model.bom.json
  RELEASE-NOTES.md
```

## 6) Windows enforcement (defense‑in‑depth)

* **Signed PowerShell manifest** for your verify‑and‑load script (loosely: `Verify-Model.ps1`).
* **WDAC policy** (Windows Defender Application Control) that allowlists:

  * Your signed loader binary (e.g., `myloader.exe`)
  * Cosign binary (optional) and hash pinning for the loader
  * Blocks modules that try to map unsigned GGUF/safetensors
* Loader adds a tiny **operator‑approval gate**:

  * Verify cosign bundles (cert chain + Rekor inclusion)
  * Recompute Merkle root from `chunks.json` before memory‑mapping
  * Check SBOM license allowlist
  * Refuse to load if **any** check fails

Pseudocode at load:

```text
verify_sig(model.gguf, model.gguf.sig, bundle)
verify_sig(merkle_root.txt, merkle_root.txt.sig, bundle)
verify_sig(provenance.json, provenance.json.sig, bundle)
recomputed_root = merklize(chunks.json, model.gguf)
assert recomputed_root == read(merkle_root.txt)
assert license_in_allowlist(parse_bom("model.bom.json"))
display_provenance_summary()
mmap(model.gguf)
```

**Across the community and recent benchmarks**, the rough pattern is now pretty consistent: when you run ~7–9B instruction‑tuned models (such as Community MPT‑7B variants, Llama‑3 8B, Qwen‑9B family) with **aggressive low‑bit quantization (q4_0 / q4_k variants)** they *often fit inside ~4–6 GB of VRAM* when you’re only using short contexts (≤8 k tokens) — fitting comfortably on many consumer GPUs. ([LocalLLM.in][1])

But as soon as you start pushing **longer contexts (32 k or 64 k)** with q4_k or q4_k_m quantizations to preserve accuracy and KV cache efficiency, the **VRAM cost climbs sharply**. KV cache grows roughly linearly with context length, meaning total memory use often lands in the **~6–8 + GB range at 32 k** and **~7.5–8 + GB or more at 64 k** even for these “small” models. ([LocalLLM.in][1])

This pattern shows up in *practical llama.cpp measurements* and is echoed across community builds:

* **Short context, aggressive 4‑bit:** fits into ~4–6 GB.
* **Long context, more efficient q4_k / q4_k_m:** often needs ~6–8GB+ just to keep the model and KV cache resident in VRAM. ([LocalLLM.in][1])

If you’re thinking about which GPU tier you need or where the *break‑points* are, those quantization and context tradeoffs are absolutely what determines when you can stay on a single consumer card versus needing 10 GB–16 GB or more. ([LocalLLM.in][1])
