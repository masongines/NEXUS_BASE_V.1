## IV. SANDBOX PROTOCOL FUNCTION — WHAT IT IS

The sandbox protocol function created in this work session is a governed method for validating whether the system can safely and truthfully describe itself **before** code rewrite or live promotion.

It is designed to answer:

- Can the snapshot system preserve source precedence?
- Can it keep conflict visible?
- Can it record exclusions without silent omission?
- Can it preserve uncertainty honestly?
- Can it stay observational rather than becoming an authority surface?
- Can terminology remain controlled under stress?
- Can correction findings be fed back into the next cycle without hidden drift?

### Compressed Definition

The sandbox protocol function is:

> a controlled simulation-and-review cycle that tests draft observability artifacts against structured scenarios, output templates, exclusion rules, uncertainty rules, and validation checklists, then produces a formal report, correction path, rerun requirement, and self-improvement signal.

## V. WHY THIS SANDBOX EXISTS

This sandbox was created because the legacy snapshot / control-surface model had known weaknesses, especially around:

- stale interface-derived state
- hidden source merging
- unlabeled uncertainty
- conflict collapse
- terminology looseness
- implementation assumptions outrunning governance logic
- visibility surfaces drifting toward implied authority
- draft-state ambiguity being mistaken for stable system state

The current sandbox work is the **infrastructure-first response** to those weaknesses.

## IX. SANDBOX VALIDATION CYCLE — HOW IT WORKS

The sandbox simulation follows this cycle:

### Phase 1 — Control Artifact Preparation
Establish:
- protocol
- spec
- alignment map
- implementation decisions
- terminology control
- lab-state source

### Phase 2 — Output Structure Preparation
Establish:
- accepted baseline
- output template
- test checklist
- exclusion and uncertainty register specs

### Phase 3 — Simulated Scenario Definition
Create scenario file that describes controlled pressure cases.

### Phase 4 — Sample Output Render
Create a representative sample snapshot output.

### Phase 5 — Validation Review
Evaluate the sample output against:
- scenarios
- template
- checklist
- exclusion rules
- uncertainty rules
- terminology rules
- precedence rules
- observational-boundary rules

### Phase 6 — Report and D&R
Produce:
- results by scenario
- findings
- corrections
- keeps
- defers
- future file implications
- self-improvement notes

### Phase 7 — Correction Cycle
Use the report to tighten the next iteration before code rewrite.

### Phase 8 — Rerun Requirement
After any material correction:
- regenerate the affected sample artifact
- rerun the relevant scenario(s)
- confirm the correction improved the system
- record whether the change reduced ambiguity, improved consistency, or revealed a new gap

**Rule:**  
No major correction is considered stable until it has been rerun against the condition that exposed it.

## XV. OVERRIDE / EXCEPTION HANDLING RULE

This section exists to prevent rigid process from becoming brittle or pseudo-authoritative.

### General Rule
No exception may silently bypass:
- source precedence
- conflict visibility
- exclusion visibility
- uncertainty handling
- observational boundary discipline
- draft/base separation

### Allowed exception posture
Exceptions may only exist as:
- explicit operator-directed exception
- clearly labeled advisory-only deviation
- review-pending draft-side workaround

### Required exception disclosures
Any exception must record:
- what was bypassed
- why the exception exists
- whether it is advisory-only or operator-directed
- what risk it introduces
- what correction or review path closes it

### Hard stop conditions
Pause continuation and escalate for review if a proposed action would:
- imply Base V1 migration
- imply implementation readiness
- imply generator rewrite authorization
- hide known uncertainty
- collapse conflict or exclusion visibility
- convert observational output into authority-like status

This sandbox protocol is not only for validation.
It is also a self-improvement engine.

Each sandbox cycle should produce findings about:

- what worked clearly
- what caused friction
- what remains underdefined
- what terminology is still weak
- what file is missing or overloaded
- what should be corrected next
- whether the system is getting more precise or merely more complex

### Sandbox refinement rule
When a weakness is discovered:
1. identify the exact issue
2. correct the issue locally
3. rerun the prior scenario or check that exposed it
4. confirm whether the correction improved the result
5. record residual weakness if any remains
6. continue only after the local correction path is understood

**Rule:**  
Self-evaluation is part of the protocol, not an optional extra.
