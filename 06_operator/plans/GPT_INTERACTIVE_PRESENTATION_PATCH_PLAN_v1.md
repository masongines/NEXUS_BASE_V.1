# GPT_INTERACTIVE_PRESENTATION_PATCH_PLAN_v1

**Classification:** Patch Plan / Non-Doctrinal  
**Authority:** Advisory / Operator-Governed  
**Status:** Active Patch Planning Artifact  
**Scope:** Define the exact patch path required to convert the current GPT interactive presentation artifact into a stronger, more current, and more trustworthy internal reference surface without turning it into a live truth source or collapsing it into the external portfolio site

---

## I. PURPOSE

This artifact defines the exact patch path for the current GPT interactive presentation artifact.

The goal is not to rebuild the artifact from scratch.

The goal is to:

1. preserve the existing shell
2. correct stale state
3. improve source tagging
4. separate internal reference logic from external portfolio logic
5. strengthen truth-boundary handling
6. fix current UI/logic weaknesses

This patch plan does not authorize:
- live runtime behavior
- live system-state authority
- dashboard promotion into canonical truth
- merging internal dashboard and public portfolio into one surface

## VII. PATCH SET C — INVENTORY AND SOURCE POSTURE

### PATCH_C_01 — COUNT SAFETY
All counts currently embedded in the top bar and data object must be reviewed.

Required rule:
- no count should appear as trusted current truth unless it is grounded in approved current source basis

### PATCH_C_02 — SOURCE TYPE TAGS
Each major panel should show one of the following:

- CANONICAL
- ACTIVE_STANDARD
- SUPPORT_RECORD
- CHECKPOINT
- OBSERVATIONAL_ONLY
- WEAKER_SURFACE
- FUTURE_ONLY
- REFERENCE_ONLY

### PATCH_C_03 — FRESHNESS TAGS
Each major section should carry a freshness tag:

- CURRENT
- STALE
- BASELINE_ONLY
- UNKNOWN
- NEEDS_REFRESH

### PATCH_C_04 — STRONGER SOURCE WARNINGS
Make weaker-surface warnings harder to miss when:
- dashboard text conflicts with registry/checkpoint truth
- observational outputs could be misread as authority
- counts are derived rather than trusted

### PATCH_C_05 — IMPLEMENTED / PLANNED / REFERENCE-ONLY SPLIT
Where relevant, each artifact/state group should explicitly identify whether it is:
- IMPLEMENTED
- PLANNED
- REFERENCE_ONLY
- FUTURE_ONLY
- DEFERRED

## VIII. PATCH SET D — TECHNICAL FIXES

### PATCH_D_01 — OBSERVATIONAL SURFACE TOGGLE
Current behavior is a no-op.

Required fix:
- make the observational-surface toggle actually hide/show designated observational sections
- use explicit section targeting rather than same-value assignment

### PATCH_D_02 — STRONGEST_ONLY_MODE
Current behavior only hides meta strips.

Required fix:
- strongest-only mode should reduce display to strongest-source surfaces only
- at minimum:
  - operator-level framing
  - canonical doctrine / registry
  - active standards
  - locked checkpoints
- weaker surfaces and observational surfaces should dim or hide

### PATCH_D_03 — FILTER MODEL
Current filtering is text-fragile.

Required fix:
- move filtering to metadata-driven logic
- assign structured tags to cards/sections
- filter by actual class/status/source-type rather than incidental text matches

### PATCH_D_04 — VIEW MODE SHALLOWNESS
Current view modes are mostly cosmetic shortcuts.

Required improvement:
- Executive mode should emphasize:
  - identity
  - authority hierarchy
  - core architecture
  - checkpoint state
  - next sequence
- Full mode should show everything
- Audit mode should:
  - expand discrepancy-heavy content
  - emphasize freshness/source warnings
  - foreground weaker-surface notices

### PATCH_D_05 — LOCAL UI STATE SAFETY
Keep local UI state persistence, but ensure it stores:
- display preferences only
- no pseudo-state
- no system-truth implications
