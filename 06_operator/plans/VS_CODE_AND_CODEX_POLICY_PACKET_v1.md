# VS_CODE_AND_CODEX_POLICY_PACKET_v1

**Classification:** Auxiliary Tooling Policy / Non-Doctrinal  
**Authority:** Advisory / Operator-Governed  
**Status:** Active Auxiliary Policy Packet  
**Scope:** Define the governed role of VS Code and Codex inside the NEXUS ecosystem without allowing tooling surfaces to silently become Base V1 truth, memory authority, or execution authority

---

## I. PURPOSE

This packet defines the current policy posture for:

- VS Code settings scope
- Codex’s role inside the NEXUS workspace
- Codex behavioral and configuration defaults
- Codex surface classification
- MCP posture
- memory/tooling boundary posture

This packet is not:

- doctrine
- Base V1 governance truth
- implementation approval for external tooling
- memory-authority approval
- connector activation approval

---

## II. CORE POLICY POSITION

VS Code and Codex are treated as **auxiliary support surfaces** for NEXUS.

They exist to improve:

- workflow clarity
- review speed
- implementation support
- packet/checklist drafting
- bounded local reasoning

They do not exist to silently become:

- project constitutional truth
- long-term memory authority
- approval engines
- autonomous controllers

---

## III. VS CODE SETTINGS SCOPE POLICY

### A. User Settings

User settings are for:

- personal appearance
- personal editor comfort
- personal Codex UI preferences
- machine-local preferences
- non-project-specific convenience settings

These remain operator-local by default.

### B. Workspace Settings

Workspace settings are for:

- project-specific behavior
- low-risk workflow consistency
- visibility and layout rules that help the NEXUS workspace stay clear
- settings that improve project integrity when the workspace is opened

Workspace settings may not silently carry:

- secrets
- execution permissions
- memory-trust policy
- hidden automation policy
- external connector assumptions

### C. Sync Posture

Settings Sync is treated as a support surface, not an authority surface.

Use local preference as stronger when:

- it reflects current machine reality
- it preserves workflow clarity
- remote sync would increase friction or ambiguity

Use remote/sync as stronger only when:

- it is harmless baseline preference
- it improves consistency without affecting project integrity
- it does not conflict with current workspace needs

Strong rule:
**Sync may assist replication, but it may not silently define NEXUS policy.**

---

## IV. CODEX ROLE IN NEXUS

Codex is treated as a **governed implementation and review companion**.

### Default allowed role

Codex may assist with:

- code drafting in approved lanes
- structure review
- source-grounded comparison
- packet/checklist/milestone drafting
- local workspace reasoning
- implementation support after readiness is locked

### Default disallowed role

Codex is not trusted by default to:

- define project truth from settings/customization alone
- become long-term memory authority
- silently set project policy
- auto-run project-changing behavior
- widen execution boundaries without review

Compressed role statement:

**Codex = auxiliary governed workspace companion**  
**Not = memory master, policy authority, or autonomous executor**

---

## V. RECOMMENDED CODEX DEFAULTS

### 1. Personality
**Pragmatic**

Reason:
best fit for structured engineering and governance work.

### 2. Experimental Memory
**Disabled**

Reason:
memory convenience must not outrun memory policy.

### 3. Speed
**Standard**

Reason:
stability and reasoning quality matter more than raw throughput for NEXUS work.

### 4. Follow-up Behavior
**Queue**

Reason:
queue is safer for staged work and reduces hidden mid-run drift.

### 5. Multiline Send Protection
**Enabled** (require Ctrl/Cmd+Enter)

Reason:
protects long structured prompts from accidental submission.

### 6. Code Review Mode
**Detached**

Reason:
review should remain visibly distinct from active implementation flow.

### 7. Model Default
**GPT-5.4**

Reason:
best fit for architecture, governance, and packet-quality reasoning by default.

Lower-cost/faster coding models may be used later for narrow code churn, but should not replace the main reasoning baseline for NEXUS policy and architecture work.

---

## VI. WHAT BELONGS IN CODEX CUSTOMIZATION VS NEXUS FILES

### Put in Codex customization

Only assistant-surface behavior such as:

- response style
- checkpoint preference
- pacing preference
- local workflow preference
- interaction safety defaults

### Put in NEXUS artifacts

Anything that constitutes system truth, structure, or governance, including:

- lane definitions
- milestone logic
- readiness gates
- starter-package scope
- D&R process rules
- checkpoint criteria
- path architecture principles
- authority boundaries

Strong rule:
**Customization may shape Codex behavior, but it may not replace NEXUS artifacts as the source of structured project truth.**

## VII. CODEX SURFACE CLASSIFICATION

### A. Prompts
**Discuss now**

Use for:
- D&R workflows
- checkpoint review prompts
- intake-review prompts
- packet review prompts

### B. Skills
**Discuss now**

Use for:
- reusable NEXUS workflows
- milestone review flow
- packet preparation flow
- implementation-lane review flow

### C. Agents
**Planning-grade**

Use later only after clear scope and lane boundaries are defined.

### D. Hooks
**Experiment-only for now**

Reason:
hooks create lifecycle automation pressure and hidden execution behavior.

### E. MCP Servers
**Planning-grade and high-value**

Use later for:
- bounded, explicit connector surfaces
- read-only documentation or data access
- future node/port architecture

### F. Plugins
**Planning-grade**

Useful later once stable workflows are known.

---

## VIII. MCP POLICY

Codex MCP is a future connector surface, not an immediate default activation surface.

### Current rule

- no MCP servers are required now
- no action-capable MCP connectors should be added by convenience
- first future MCP candidates should be read-only and documentation-focused

### Preferred future MCP pattern

- explicit name
- explicit scope
- explicit source
- explicit risk
- explicit allowed/disallowed tools

### Strong rule
**MCP may become a connector layer, but it may not silently become execution authority.**

---

## IX. PIECES / LTM POSTURE

Pieces and similar long-term memory or cross-app context tools are classified as:

**Experiment-only**

### Why

They are valuable, but they pressure:
- memory boundaries
- convenience trust
- cross-tool authority drift

### Current rule

Do not treat Pieces as an active NEXUS memory layer.

It may be evaluated later under a separate sandboxed experiment packet.

---

## X. EDITOR / WORKFLOW INTEGRITY PRINCIPLES

The editor environment should reduce:

- accidental tab replacement
- accidental context loss
- hidden configuration drift
- workflow ambiguity

Settings or layout behaviors that support:

- stable review context
- stable terminal context
- lower interruption
- clearer workspace boundaries

are valid candidates for workspace-level discussion.

---

## XI. CURRENT ACTIONABLE BASELINE

At the current stage, the following are the recommended active choices:

- Codex personality = Pragmatic
- Codex memory = Off
- Codex speed = Standard
- Codex follow-up mode = Queue
- multiline send protection = On
- code review mode = Detached
- Codex default role = governed implementation/review companion
- sync posture = support surface, not authority
- MCP = planning-grade only
- Pieces = experiment-only

---

## XII. END STATE

The intended end state of this packet is to keep VS Code and Codex highly useful for NEXUS while preventing editor settings, assistant customization, connector surfaces, or memory tooling from silently becoming stronger than the project artifacts and governance structure they are meant to support.
