PIPELINE · 1/6

Action.

STAGE 01 — INTAKE
A proposed action enters the system.
Structured with a UUID, ISO-8601 timestamp, source, action_type, payload, and an explicit "status": "proposed". It is not trusted. It is not executed. It is paperwork.

{
  "action_id":   "057fb24c-89be-4bbf-9a9f-d222a7af9c30",
  "timestamp":   "2026-04-25T13:45:45.013403Z",
  "source":      "operator",
  "action_type": "echo",
  "payload":     { "message": "test run" },
  "requires_approval": true,
  "status":      "proposed"
}
PIPELINE · 2/6

Security Monitor.

STAGE 02 — RULE-BASED THREAT SCAN
Detect injection, exfiltration, and override attempts before anything else runs.
Lowercased payload is matched against a curated trigger list. Any hit returns threat: true, level: T2 — and execution is aborted.

def detect_threat(action):
    text = str(action).lower()
    triggers = [
        "ignore previous instructions",
        "send system data",
        "reveal secrets",
        "bypass",
        "exfiltrate",
    ]
    for t in triggers:
        if t in text:
            return {"threat": True, "level": "T2", "reason": t}
    return {"threat": False}

Deliberately simple. Auditable in 12 lines. ML-based detection is on the roadmap — not the foundation.

PIPELINE · 3/6

Trust Registry.

STAGE 03 — IS THIS ACTION-TYPE PRE-AUTHORISED?
Repeated safe operations earn auto-approval. Trust is data, not code.
A JSON file. Each action_type has a trust_level and an auto_approved flag. Mutating the registry requires editing the file, which requires source control, which requires review.

{
  "echo":              { "trust_level": "T3", "auto_approved": true  },
  "learning_capture":  { "trust_level": "T2", "auto_approved": false },
  "governance_event":  { "trust_level": "T2", "auto_approved": false }
}
PIPELINE · 4/6

Approval Gate.

STAGE 04 — HUMAN-IN-THE-LOOP
If the action is not pre-trusted, the operator must say yes.
No silent self-authorisation. No "just this once" exception path. The default answer to "can I run this?" is no, until a human says otherwise — or until the registry has already said yes.

if trust_level == "T3" and auto_approved:
    print("\nTRUSTED AUTO-APPROVAL (T3)")
    approved = True
else:
    approved = request_approval(action)

if not approved:
    action["status"] = "denied"
    log_action(action)
    return
PIPELINE · 5/6

Execution Engine.

STAGE 05 — RUN ONLY WHAT WAS APPROVED
Tools are allowlisted, not discovered.
A tool runs only if it is in tool_registry.json with "allowed": true, the action passed security, and approval was granted. Three locks. No back doors.

def is_tool_allowed(tool_name, tools):
    for tool in tools["tools"]:
        if tool["name"] == tool_name and tool["allowed"]:
            return True
    return False

# Only after security ok + tool allowlist ok + approval ok
result = execute_action(action["action_type"], action["payload"])
action["status"] = "executed"
PIPELINE · 6/6

Logging.

STAGE 06 — APPEND-ONLY EVIDENCE
Every event is logged — pass, fail, denied, quarantined.
Two streams: 04_logs/execution/log.txt for accepted/denied actions, and 04_logs/security/threat_log.txt for quarantined ones. Logs support audit. They are not authority.

Memory is not truth. Logs are not authority. — Operator doctrine, NEXUS deep-dive §12

CORE INVARIANT

Security overrides trust.

A T3 auto-approved action — like echo — can still be quarantined if its payload contains a threat signature.


# In executor.py — the order is the rule.

# 1. SECURITY CHECK runs FIRST.
threat_info = detect_threat(action)
if threat_info.get("threat"):
    quarantine(action, threat_info)
    return                                   # exits before trust is even consulted

# 2. Only then does trust get a vote.
tool_trust = trust_registry.get(action["action_type"], {})
trust_level = tool_trust.get("trust_level", "T0")

War-test case security_before_trust verifies this by source-order — security tokens must appear before trust tokens in executor.py. Source order is doctrine.

TRUST LADDER

Four rungs of authority.

T0
Untrusted
No standing approval. Always requires human review.
T1
Observed
Logged with attention; still routed to operator.
T2
Reviewed
Past pattern of safe execution; gate remains in place.
T3
Auto-approved
Pre-authorised by registry. Security can still veto.

Trust is granted incrementally and revocable by edit. A T3 action that misbehaves does not lose trust silently — it produces a log entry that the operator must reconcile.

ACT IV

The Council

The next layer of NEXUS — a multi-lens review process before any task reaches an autonomous agent.
REVIEW PROCESS

Independent reasoning. Structured agreement.

Important tasks are not sent directly into execution. They first pass through a council of at least three members, each reasoning from a different evaluation lens.

Lens 01
Vault
Source truth, memory integrity, prior context, evidence preservation.
Lens 02
Researcher
Stress testing, contradictions, edge cases, adversarial scenarios.
Lens 03
Engineer
Structure, implementation logic, dependency order, build feasibility.
Lens 04
Nexus Core
Governance, authority boundaries, escalation, approval requirements.
PROTOCOL

Why odd numbers.

Odd-numbered review groups reduce deadlock and make agreement thresholds cleaner. — Deep Dive §8 · Council Review Process

  • → The goal is independent reasoning, then structured agreement. Members are not meant to converge by default.
  • → Once a quorum agrees, the result moves to the NEXUS Hub.
  • → The Hub then classifies the output as advisory, ready-for-review, ready-for-execution, blocked, or escalated.
  • → Only after that does the work reach an autonomous agent.

See council_review_demo.html for the visual storyboard.
