COUNCIL · STORYBOARD

One action. Four lenses. Structured agreement.

This is a worked example: a non-trivial governance request travels through the NEXUS review council before it is allowed to reach an autonomous agent. Each panel below is a frame in the storyboard.

FRAME 01 · INTAKE

The operator submits an action.

A non-trivial action arrives. It is more than a routine echo — it requests a state change. The Hub does not auto-execute. It routes the action to the council.
action_id e91d4b87-2c0a-4a1d-b6f1-0c00f2f9a3b1 timestamp 2026-05-08T14:22:01Z source operator action_type governance_event payload.subject"Promote echo trust_level T3 -> T2" payload.reason "observed unexpected log-volume spike" requires_approvaltrue status proposed
FRAME 02 · PRECONDITION

Security clears the action for review.

Before the council convenes, the rule-based monitor scans the payload. The action contains no injection markers, no bypass language, no exfiltration keywords. It passes. Security always runs first.
detect_threat() -> { "threat": false } trust_lookup governance_event -> T2 · auto_approved=false routed_to council_review

FRAME 05 · OPERATOR DECISION

The operator presses the gate.

The Engineer's conditional ("re-run war test") is folded into the execution package. The operator approves with the war-test re-run as a hard prerequisite. The action becomes execution-ready — but only because a human said yes.

FRAME 06 · DOCTRINE

What never changed during this review.

No matter what the council recommended, no matter what the operator decided, the same six rules held throughout the process.
-> No self-authorisation
-> No hidden execution
-> Memory is not truth
-> Logs are not authority
-> Security overrides trust
-> Operator authority remains final
Continues into the execution pipeline. Once the operator approves and the war test re-passes, the action re-enters executor.py at the Security stage — not as a council output, but as a new "proposed" action with full provenance attached.
